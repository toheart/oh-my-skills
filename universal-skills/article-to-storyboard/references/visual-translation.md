# Visual Translation

## Purpose

Use this reference when turning a written idea into a visual plan.
This is the stage where many video prompts become generic.

## Principle

Do not ask "what cool image fits this topic?"
Ask "what visual form best communicates this scene's role in the argument?"

## Start With Visual Role

Common roles:

- `thesis`
- `evidence`
- `contrast`
- `process`
- `example`
- `summary`

The role determines the most useful visual language.

## Then Choose Visual Type

Suggested mappings:

- thesis -> kinetic typography or bold statement card
- evidence -> chart, callout, or diagram
- contrast -> split layout, opposing labels, before/after
- process -> flow diagram, staged build, timeline
- example -> image-led scene or specific case card
- summary -> recap list or closing statement

## Literal vs Metaphorical

Decide explicitly whether the scene should be:

- literal
- metaphorical
- mixed

If it is metaphorical, explain the abstraction in one sentence.
Do not assume the renderer will infer the right kind of metaphor.

## Use Avoid Lists Aggressively

When the source is abstract, generic imagery appears quickly.
Default candidates to avoid include:

- robot faces
- floating brains
- code rain
- futuristic city skylines
- holographic UI wallpaper
- unrelated stock-office footage

## Good Visual Prompts

Good prompts name:

- the communication goal
- the composition style
- the elements that matter
- what to avoid

Bad prompts are just topic labels with visual adjectives.

Bad:

- cinematic AI future scene with dramatic light

Better:

- restrained editorial composition showing task handoff friction, review loops, and blocked workflow states; no robot, no code rain, no abstract cyber backdrop

## Review Questions

Before finalizing a scene, ask:

- Does this visual explain the scene or merely decorate it?
- Could a reviewer trace it back to the source?
- Is it specific enough to avoid generic substitutes?
- Does the motion reinforce meaning or just add activity?
